# /ship — Пост-чейндж воркфлоу

Выполни следующий воркфлоу после внесения изменений в код. Действуй строго по шагам, не пропуская ни одного.

## Шаги

### Шаг 1 — Обновить тесты

Проанализируй изменённые файлы (через `git diff --name-only HEAD` или список из контекста).
Для каждого изменённого модуля:
- Найди соответствующие тестовые файлы (паттерны: `*.test.ts`, `*.test.tsx`, `*.test.py`, `test_*.py`).
- Обнови или добавь тесты, отражающие новое поведение.
- Если тестов нет — создай минимально необходимые.

### Шаг 2 — Запустить тесты

Определи команду для запуска тестов по структуре проекта:
- Frontend (`apps/frontend`): `cd apps/frontend && npm test -- --watchAll=false` или `npx vitest run`
- Backend (`apps/api`): `cd apps/api && python -m pytest -v`
- Если есть `Makefile` или `package.json` скрипты — используй их.

Запусти тесты и покажи вывод.

### Шаг 3 — Исправить ошибки

Если тесты упали:
- Прочитай каждую ошибку.
- Найди первопричину (не симптом).
- Внеси исправление в **код или тест** (в зависимости от того, что неверно).
- Повтори Шаг 2.
- Повторяй до тех пор, пока все тесты не пройдут (максимум 3 итерации, после чего сообщи о проблеме пользователю).

### Шаг 4 — Ревью изменений

Запусти агента-ревьюера. Передай ему:
- `git diff HEAD` (или diff конкретных файлов)
- Контекст задачи из текущего разговора

Агент должен проверить:
- Корректность логики
- Потенциальные баги или edge cases
- Нарушения архитектурных паттернов проекта (см. CLAUDE.md)
- Безопасность (инъекции, утечки токенов, открытые эндпоинты)
- Читаемость и стиль

Выведи замечания ревьюера. Если есть критические — исправь их и вернись к Шагу 2.

### Шаг 5 — Предложить коммит-сообщение

Сформируй коммит-сообщение по правилам:
- Первая строка: `<тип>(<scope>): <краткое описание>` (не длиннее 72 символов)
- Типы: `feat`, `fix`, `refactor`, `test`, `docs`, `chore`, `style`, `perf`
- Пустая строка, затем тело (bullet-list с ключевыми изменениями, если их больше одного)
- Язык: русский или английский — такой же, как в последних коммитах репозитория

Выведи итоговое сообщение в блоке кода, готовое для копирования.

---

> **Примечание:** Не делай `git commit` автоматически — только предложи сообщение.
